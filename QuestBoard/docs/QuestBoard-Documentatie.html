1. Projectoverzicht

QuestBoard is een gamified freelance platform waar taken worden gemodelleerd als RPG-quests. Freelancers verdienen XP, gold en badges door quests te voltooien. Het project demonstreert 6 GoF design patterns in een werkende ASP.NET Core Web API.

2. Strategy Pattern

GoF CategorieBehavioral
Punten4
Bestanden7 bronbestanden, 6 tests

2.1 Probleem

QuestBoard heeft meerdere prijsberekeningen (vast, veiling, dynamisch) en matchmaking-algoritmes (skill-based, rating-based). Zonder pattern zou dit leiden tot lange if/else-ketens in de service laag die moeilijk uitbreidbaar zijn. Bij elke nieuw pricing model of matchmaking strategie moet bestaande code worden aangepast.

2.2 Oplossing

Het Strategy pattern definieert een familie van algoritmes, kapselt elk in een eigen klasse, en maakt ze uitwisselbaar. De client (QuestService) kiest runtime welke strategie wordt gebruikt zonder de algoritme-implementatie te kennen.

2.3 Class Diagram

+---------------------+ | IPricingStrategy |<----- QuestService +---------------------+ (gebruikt via DI) | +Name: string | | +CalculatePrice() | +---------------------+ ^ ^ ^ | | | +------+ +-------+ +--------+ |Fixed | |Auction | |Dynamic | +------+ +-------+ +--------+ |100% | |80% | |factor | +------+ +-------+ |+urgency| +--------+ +----------------------+ | IMatchmakingStrategy |<----- QuestService +----------------------+ | +Name: string | | +FindMatches() | +----------------------+ ^ ^ | | +----------+ +-----------+ |SkillBased| |RatingBased| +----------+ +-----------+

2.4 Implementatie

// Interface - definieert het contract voor alle pricing strategieen
public interface IPricingStrategy
{
    string Name { get; }
    decimal CalculatePrice(decimal basePrice, int difficulty, DateTime? deadline);
}

// Concrete strategy - DynamicPricingStrategy past prijs aan op moeilijkheid en deadline
public class DynamicPricingStrategy : IPricingStrategy
{
    public string Name => "Dynamic";

    private static readonly Dictionary<int, decimal> DifficultyMultipliers = new()
    {
        [0] = 0.8m,  // Easy
        [1] = 1.0m,  // Medium
        [2] = 1.3m,  // Hard
        [3] = 1.6m,  // Epic
        [4] = 2.0m   // Legendary
    };

    public decimal CalculatePrice(decimal basePrice, int difficulty, DateTime? deadline)
    {
        var multiplier = DifficultyMultipliers.GetValueOrDefault(difficulty, 1.0m);
        var price = basePrice * multiplier;

        // Urgency surcharge: +25% als deadline binnen 3 dagen
        if (deadline.HasValue && (deadline.Value - DateTime.UtcNow).TotalDays <= 3)
            price *= 1.25m;

        return price;
    }
}

// Gebruik in QuestService - strategy wordt runtime geselecteerd
var strategy = _pricingStrategies
    .FirstOrDefault(s => s.Name.Equals(dto.PricingType.ToString(),
        StringComparison.OrdinalIgnoreCase))
    ?? _pricingStrategies.First();

var calculatedGold = strategy.CalculatePrice(dto.BaseGold, (int)dto.Difficulty, dto.Deadline);

2.5 Voordelen en Nadelen

  • + Open/Closed Principle: nieuwe strategie toevoegen zonder bestaande code te wijzigen
  • + Strategieen zijn onafhankelijk testbaar
  • + Runtime wisselen van algoritme mogelijk
  • - Meer klassen nodig dan een simpele if/else
  • - Client moet weten welke strategieen bestaan

2.6 Alternatieven

AlternatiefWaarom niet gekozen
If/else of switch in serviceSchendt Open/Closed Principle. Elke nieuwe pricing vereist aanpassing van bestaande code.
Template MethodVereist overerving i.p.v. compositie. Minder flexibel: kan niet runtime wisselen.
Command PatternTe zwaar voor simpele berekeningen. Command is beter voor undo/redo scenarios.

2.7 Uitbreidbaarheid

Een nieuwe pricing strategie toevoegen (bijv. SubscriptionPricingStrategy) vereist slechts:

  1. Nieuwe klasse aanmaken die IPricingStrategy implementeert
  2. Registreren in DI container: builder.Services.AddSingleton<IPricingStrategy, SubscriptionPricingStrategy>()

Geen bestaande code hoeft te worden aangepast.

3. Flyweight Pattern

GoF CategorieStructural
Punten4
Bestanden4 bronbestanden, 5 tests

3.1 Probleem

In QuestBoard hebben veel freelancers dezelfde skills (bijv. "C#", "React") en verdienen dezelfde badges. Zonder Flyweight zou elke freelancer een eigen kopie van elk skill/badge-object in geheugen hebben. Bij 1000 freelancers met gemiddeld 5 skills = 5000 skill-objecten, terwijl er maar 15 unieke skills zijn.

3.2 Oplossing

Het Flyweight pattern deelt objecten die veel voorkomen via een factory pool. De SkillFactory en BadgeFactory garanderen dat er per unieke skill/badge slechts 1 object in geheugen bestaat. Meerdere freelancers verwijzen naar hetzelfde object.

3.3 Class Diagram

+------------------+ +------------------+ | SkillFactory | | SkillFlyweight | | (Singleton) |-------->| (Immutable) | +------------------+ +------------------+ | -_pool: Dict | | +Name: string | | +GetSkill() | | +Category: string| | +GetSkillCount() | | +IconUrl: string | | +GetAllSkills() | +------------------+ +------------------+ +------------------+ +------------------+ | BadgeFactory | | BadgeFlyweight | | (Singleton) |-------->| (Immutable) | +------------------+ +------------------+ | -_pool: Dict | | +Name: string | | +GetBadge() | | +Description | | +GetBadgeCount() | | +Tier: string | | +GetAllBadges() | | +IconUrl: string | +------------------+ +------------------+ Freelancer A --\ Freelancer B ---+--> SkillFlyweight("C#") (zelfde object) Freelancer C --/

3.4 Implementatie

// Flyweight object - immutable, gedeeld via factory
public sealed class SkillFlyweight
{
    public string Name { get; }
    public string Category { get; }
    public string IconUrl { get; }

    public SkillFlyweight(string name, string category, string iconUrl)
    {
        Name = name ?? throw new ArgumentNullException(nameof(name));
        Category = category ?? throw new ArgumentNullException(nameof(category));
        IconUrl = iconUrl ?? throw new ArgumentNullException(nameof(iconUrl));
    }
}

// Factory met singleton pool - garandeert 1 object per unieke naam
public sealed class SkillFactory
{
    private static readonly Lazy<SkillFactory> _instance = new(() => new SkillFactory());
    private readonly ConcurrentDictionary<string, SkillFlyweight> _pool = new();

    private SkillFactory() { }
    public static SkillFactory Instance => _instance.Value;

    public SkillFlyweight GetSkill(string name, string category = "General",
        string iconUrl = "")
    {
        return _pool.GetOrAdd(name, key => new SkillFlyweight(key, category, iconUrl));
    }
}

// Bewijs dat objecten gedeeld worden (unit test)
var skill1 = factory.GetSkill("C#");
var skill2 = factory.GetSkill("C#");
Assert.True(ReferenceEquals(skill1, skill2)); // Zelfde object in geheugen

3.5 Voordelen en Nadelen

  • + Significant minder geheugengebruik bij veel gedeelde objecten
  • + ConcurrentDictionary maakt het thread-safe voor web requests
  • + Immutability voorkomt onbedoelde wijzigingen aan gedeelde state
  • - Complexer dan simpele objecten aanmaken
  • - Intrinsic/extrinsic state scheiding moet goed doordacht zijn

3.6 Alternatieven

AlternatiefWaarom niet gekozen
Gewone objecten (geen sharing)Werkt, maar verspilt geheugen bij veel duplicaten. Niet schaalbaar.
String interningAlleen voor strings, niet voor objecten met meerdere properties.
Object Pool patternObject Pool is voor hergebruik van dure objecten (DB connections). Flyweight is voor sharing van identieke read-only objecten.

3.7 Uitbreidbaarheid

Nieuwe flyweight types toevoegen (bijv. CertificationFlyweight + CertificationFactory) volgt exact hetzelfde patroon. De /api/debug/flyweight-stats endpoint toont realtime hoeveel objecten in de pools zitten.

4. Decorator Pattern

GoF CategorieStructural
Punten4
Bestanden6 bronbestanden, 7 tests

4.1 Probleem

Quests kunnen modifiers hebben: urgent (hogere beloning), featured (extra XP), team (meerdere personen). Deze modifiers moeten combineerbaar zijn: een quest kan urgent EN featured EN team zijn. Met subclassing zou dit leiden tot een class-explosie: UrgentQuest, FeaturedQuest, UrgentFeaturedQuest, UrgentTeamQuest, etc.

4.2 Oplossing

Het Decorator pattern wikkelt quest behavior in lagen. Elke decorator voegt specifiek gedrag toe en delegeert de rest naar het inner object. Decorators zijn stapelbaar: new UrgentDecorator(new FeaturedDecorator(new BaseQuestBehavior())).

4.3 Class Diagram

+------------------+ | IQuestBehavior |<-----------------+ +------------------+ | | +CalculateGold() | | | +CalculateXp() | | | +GetTags() | | | +GetDescription()| | | +GetMaxTeamSize()| | +------------------+ | ^ ^ | | | | +--------+ +----------------+ | |BaseQuest| |QuestDecorator |-------- + |Behavior | |(abstract) | _inner +--------+ +----------------+ |passthru| | #_inner | +--------+ +----------------+ ^ ^ ^ | | | +--------+ +------+ +------+ |Urgent | |Featured| |Team | |Decorator| |Decorator| |Decorator| +--------+ +------+ +------+ |gold*1.5 | |gold*1.2| |gold*N | |xp*2 | |xp+50 | |xp*0.8 | +--------+ +------+ +------+ Stacking: Urgent(Featured(Base)) -> gold = base * 1.2 * 1.5

4.4 Implementatie

// Interface
public interface IQuestBehavior
{
    decimal CalculateGold(decimal baseGold);
    int CalculateXp(int baseXp);
    List<string> GetTags();
    string GetDescription(string baseDescription);
}

// Abstract decorator - delegeert alles naar inner
public abstract class QuestDecorator : IQuestBehavior
{
    protected readonly IQuestBehavior _inner;
    protected QuestDecorator(IQuestBehavior inner) => _inner = inner;

    public virtual decimal CalculateGold(decimal baseGold) => _inner.CalculateGold(baseGold);
    public virtual int CalculateXp(int baseXp) => _inner.CalculateXp(baseXp);
    // ... andere methods delegeren naar _inner
}

// Concrete decorator - voegt urgency premium toe
public class UrgentQuestDecorator : QuestDecorator
{
    public UrgentQuestDecorator(IQuestBehavior inner) : base(inner) { }

    public override decimal CalculateGold(decimal baseGold) =>
        _inner.CalculateGold(baseGold) * 1.5m;

    public override int CalculateXp(int baseXp) =>
        _inner.CalculateXp(baseXp) * 2;
}

// Stacking in QuestService - decorators worden dynamisch gestapeld
IQuestBehavior behavior = new BaseQuestBehavior();
if (quest.IsUrgent)   behavior = new UrgentQuestDecorator(behavior);
if (quest.IsFeatured) behavior = new FeaturedQuestDecorator(behavior);
if (quest.IsTeamQuest) behavior = new TeamQuestDecorator(behavior, quest.MaxTeamSize);

4.5 Voordelen en Nadelen

  • + Modifiers vrij combineerbaar zonder class-explosie
  • + Single Responsibility: elke decorator doet 1 ding
  • + Open/Closed: nieuwe modifier = nieuwe decorator klasse
  • - Veel kleine objecten door de wrapping-lagen
  • - Volgorde van decorators kan het resultaat beinvloeden

4.6 Alternatieven

AlternatiefWaarom niet gekozen
Subclassing per combinatieClass-explosie: 3 modifiers = 7 combinaties (2^3 - 1). Niet schaalbaar.
If/else flags in Quest klasseSchendt Single Responsibility. Alle modifier-logica in 1 klasse.
Strategy PatternStrategy wisselt het hele algoritme. Decorator stapelt aanpassingen bovenop elkaar.

4.7 Uitbreidbaarheid

Een nieuwe modifier (bijv. SeasonalQuestDecorator met holiday bonus) vereist alleen een nieuwe klasse die QuestDecorator extend. Bestaande decorators en de QuestService hoeven niet te worden aangepast.

5. Bridge Pattern

GoF CategorieStructural
Punten5
Bestanden7 bronbestanden, 5 tests

5.1 Probleem

QuestBoard heeft 2 soorten notificaties (QuestAlert, AchievementAlert) en 3 verzendkanalen (Email, Push, Webhook). Zonder Bridge zou elke combinatie een aparte klasse vereisen: QuestAlertEmail, QuestAlertPush, AchievementAlertWebhook, etc. Dat zijn 2 x 3 = 6 klassen, en bij een nieuw kanaal of type groeit dit exponentieel.

5.2 Oplossing

Het Bridge pattern scheidt de abstractie (wat voor notificatie) van de implementatie (hoe te versturen). Beide hierarchieen kunnen onafhankelijk uitgebreid worden. Een QuestAlert kan met elke INotificationSender werken zonder de specifieke sender te kennen.

5.3 Class Diagram

Abstractie Implementatie +-------------------+ +---------------------+ | Notification | | INotificationSender | | (abstract) |-------------->| (interface) | +-------------------+ _sender +---------------------+ | +Notify() | | +Channel: string | | +SenderChannel | | +Send() | +-------------------+ +---------------------+ ^ ^ ^ ^ ^ | | | | | +--------+ +----------+ +-----+ +----+ +-------+ |Quest | |Achievement| |Email| |Push| |Webhook| |Alert | |Alert | |Sender| |Sender| |Sender| +--------+ +----------+ +-----+ +----+ +-------+ Combinaties: 2 x 3 = 6 mogelijkheden met slechts 5 klassen (i.p.v. 6 klassen zonder Bridge)

5.4 Implementatie

// Implementor interface
public interface INotificationSender
{
    string Channel { get; }
    string Send(string recipient, string subject, string body);
}

// Concrete implementor
public class EmailSender : INotificationSender
{
    public string Channel => "Email";
    public string Send(string recipient, string subject, string body) =>
        $"[EMAIL] To: {recipient} | Subject: {subject} | Body: {body}";
}

// Abstractie
public abstract class Notification
{
    protected readonly INotificationSender _sender;
    protected Notification(INotificationSender sender) => _sender = sender;
    public abstract string Notify(string recipient);
}

// Refined abstractie
public class QuestAlert : Notification
{
    public string QuestTitle { get; }
    public string QuestStatus { get; }

    public QuestAlert(INotificationSender sender, string questTitle, string questStatus)
        : base(sender) { QuestTitle = questTitle; QuestStatus = questStatus; }

    public override string Notify(string recipient) =>
        _sender.Send(recipient,
            $"Quest Update: {QuestTitle}",
            $"Quest '{QuestTitle}' status changed to: {QuestStatus}");
}

// Gebruik - zelfde alert, verschillende senders
var emailAlert = new QuestAlert(new EmailSender(), "Build API", "Completed");
var pushAlert  = new QuestAlert(new PushNotificationSender(), "Build API", "Completed");
// Beide werken, andere output

5.5 Voordelen en Nadelen

  • + Voorkomt class-explosie bij meerdere dimensies
  • + Abstractie en implementatie onafhankelijk uitbreidbaar
  • + Runtime wisselen van sender mogelijk
  • - Extra abstractielaag kan overkill zijn bij weinig combinaties
  • - Indirectie maakt debugging complexer

5.6 Alternatieven

AlternatiefWaarom niet gekozen
Overerving per combinatieClass-explosie: N types x M kanalen klassen. Niet schaalbaar.
Strategy PatternStrategy wisselt 1 dimensie (het algoritme). Bridge scheidt 2 onafhankelijke hierarchieen.
Abstract FactoryAbstract Factory creert families van objecten. Bridge koppelt abstractie aan implementatie.

5.7 Uitbreidbaarheid

Een nieuw kanaal (bijv. SmsSender) of een nieuw notificatietype (bijv. PaymentAlert) toevoegen vereist slechts 1 nieuwe klasse. Alle bestaande combinaties blijven werken.

6. Interpreter Pattern

GoF CategorieBehavioral
Punten5
Bestanden6 bronbestanden, 9 tests

6.1 Probleem

Achievement-regels moeten flexibel definieerbaar zijn zonder code-aanpassingen. Voorbeelden: "minimaal 10 quests voltooid", "rating boven 4.5 en level 5+". Hardcoded if-statements vereisen een nieuwe deployment voor elke nieuwe achievement. We willen regels als data opslaan en runtime evalueren.

6.2 Oplossing

Het Interpreter pattern definieert een grammatica voor achievement-regels als DSL (Domain Specific Language). De AchievementRuleParser parsed strings zoals "quests.completed >= 10 and rating.avg > 4.5" naar een expression tree die ge-evalueerd wordt tegen freelancer stats.

6.3 Class Diagram

+------------------+ +---------------------+ | IExpression | | AchievementContext | +------------------+ +---------------------+ | +Interpret(ctx) | | -_properties: Dict | +------------------+ | +GetValue(key) | ^ ^ ^ +---------------------+ | | | +------+ +---+ +----+ +----------------------+ |Compar| |And| |Or | | AchievementRuleParser| |ison | |Exp| |Exp | +----------------------+ +------+ +---+ +----+ | +Parse(string): IExp | |prop | |L,R| |L,R | +----------------------+ |op | +---+ +----+ |value | DSL Grammatica: +------+ rule = or_group or_group = and_group (" or " and_group)* and_group = comparison (" and " comparison)* comparison = property operator value

6.4 Implementatie

// Terminal expression - vergelijkt een property met een waarde
public class ComparisonExpression : IExpression
{
    private readonly string _property;
    private readonly string _operator;
    private readonly double _value;

    public bool Interpret(AchievementContext context)
    {
        var actual = context.GetValue(_property);
        return _operator switch
        {
            ">"  => actual > _value,
            ">=" => actual >= _value,
            "<"  => actual < _value,
            "<=" => actual <= _value,
            "==" => Math.Abs(actual - _value) < 0.0001,
            _ => throw new InvalidOperationException($"Unknown operator: {_operator}")
        };
    }
}

// Non-terminal expression - logische AND
public class AndExpression : IExpression
{
    private readonly IExpression _left, _right;
    public bool Interpret(AchievementContext context) =>
        _left.Interpret(context) && _right.Interpret(context);
}

// Parser - zet string om naar expression tree
// "quests.completed >= 10 and rating.avg > 4.5"
//   -> AndExpression(
//        ComparisonExpression("quests.completed", ">=", 10),
//        ComparisonExpression("rating.avg", ">", 4.5))
public static class AchievementRuleParser
{
    public static IExpression Parse(string rule)
    {
        // 1. Split op " or " (lagere precedentie)
        var orParts = rule.Split(" or ");
        var orExpressions = orParts.Select(ParseAndGroup).ToArray();
        // 2. Bouw OR tree
        IExpression result = orExpressions[0];
        for (var i = 1; i < orExpressions.Length; i++)
            result = new OrExpression(result, orExpressions[i]);
        return result;
    }
}

6.5 Voordelen en Nadelen

  • + Achievement regels zijn data, niet code: aanpasbaar zonder recompilatie
  • + Nieuwe operators/properties toevoegen is eenvoudig
  • + Expression tree is composable: complexe regels uit simpele delen
  • - Parser schrijven is complex en foutgevoelig
  • - Geen ondersteuning voor haakjes (bewuste vereenvoudiging)
  • - Performance: elke evaluatie parsed en bouwt een tree

6.6 Alternatieven

AlternatiefWaarom niet gekozen
Hardcoded if/else per achievementNiet flexibel. Nieuwe achievement = code wijziging + deployment.
Scripting engine (Roslyn, Lua)Te zwaar voor simpele regels. Beveiligingsrisico bij user input.
Specification PatternVergelijkbaar maar minder geschikt voor string-based regels. Specification werkt beter met typed predicates.

6.7 Uitbreidbaarheid

Nieuwe operators (bijv. "contains", "between") of nieuwe expression types (bijv. NotExpression) toevoegen door een nieuwe IExpression implementatie te maken en de parser uit te breiden.

7. Observer Pattern

GoF CategorieBehavioral
Punten4
Bestanden7 bronbestanden, 6 tests

7.1 Probleem

Wanneer een quest wordt voltooid, moeten meerdere onafhankelijke acties plaatsvinden: XP berekenen, achievements checken, leaderboard bijwerken, notificaties versturen. Zonder Observer zou de QuestService al deze logica direct moeten aanroepen, wat tight coupling creert. Bij elke nieuwe actie moet de service worden aangepast.

7.2 Oplossing

Het Observer pattern definieert een one-to-many relatie: de QuestEventPublisher notifieert alle geregistreerde subscribers wanneer een quest wordt voltooid. Subscribers zijn losgekoppeld van de publisher en van elkaar.

7.3 Class Diagram

+------------------------+ +-------------------------+ | IQuestEventPublisher | | IQuestEventSubscriber | +------------------------+ +-------------------------+ | +Subscribe(sub) | | +Name: string | | +Unsubscribe(sub) | | +OnQuestCompleted(evt) | | +Notify(evt) | +-------------------------+ | +GetSubscribers() | ^ ^ ^ +------------------------+ | | | | +--------+ +----------+ +----------+ v |XpCalc | |Achievement| |Leaderboard| +------------------------+ |Sub | |CheckerSub | |UpdaterSub | | QuestEventPublisher | +--------+ +----------+ +----------+ +------------------------+ |+XP/Gold| |+DSL eval | |+score | | -_subscribers: List | |+level | |+badges | |+ranking | +------------------------+ +--------+ +----------+ +----------+ QuestService --> Publisher.Notify(evt) --> alle subscribers

7.4 Implementatie

// Event data
public class QuestCompletedEvent
{
    public Quest Quest { get; }
    public Freelancer Freelancer { get; }
    public decimal GoldEarned { get; }
    public int XpEarned { get; }
}

// Publisher - beheert subscribers en stuurt events
public class QuestEventPublisher : IQuestEventPublisher
{
    private readonly List<IQuestEventSubscriber> _subscribers = new();

    public void Subscribe(IQuestEventSubscriber subscriber) => _subscribers.Add(subscriber);
    public void Unsubscribe(IQuestEventSubscriber subscriber) => _subscribers.Remove(subscriber);

    public void Notify(QuestCompletedEvent evt)
    {
        foreach (var subscriber in _subscribers)
            subscriber.OnQuestCompleted(evt);
    }
}

// Subscriber - reageert op quest completion
public class XpCalculatorSubscriber : IQuestEventSubscriber
{
    public string Name => "XpCalculator";

    public void OnQuestCompleted(QuestCompletedEvent evt)
    {
        evt.Freelancer.Xp += evt.XpEarned;
        evt.Freelancer.Gold += evt.GoldEarned;
        evt.Freelancer.Level = 1 + (evt.Freelancer.Xp / 1000);
        evt.Freelancer.QuestsCompleted++;
    }
}

// Wiring in Program.cs
publisher.Subscribe(app.Services.GetRequiredService<XpCalculatorSubscriber>());
publisher.Subscribe(app.Services.GetRequiredService<AchievementCheckerSubscriber>());
publisher.Subscribe(app.Services.GetRequiredService<LeaderboardUpdaterSubscriber>());

7.5 Voordelen en Nadelen

  • + Losse koppeling: publisher kent subscribers niet bij naam
  • + Nieuwe subscriber toevoegen zonder bestaande code aan te passen
  • + Subscribers zijn onafhankelijk testbaar
  • - Volgorde van notificatie kan relevant zijn (XP moet voor achievements)
  • - Debugging lastig: event flow niet direct zichtbaar in code

7.6 Alternatieven

AlternatiefWaarom niet gekozen
C# events/delegatesWerkt, maar minder expliciet en moeilijker te documenteren voor een schoolopdracht. Custom interfaces tonen het pattern duidelijker.
Mediator (MediatR)Externe dependency. Observer is eenvoudiger en toont het pattern puurder.
Directe method calls in serviceTight coupling. Elke nieuwe actie vereist aanpassing van QuestService.

7.7 Uitbreidbaarheid

Een nieuwe subscriber (bijv. EmailNotificationSubscriber, AnalyticsSubscriber) toevoegen vereist:

  1. Nieuwe klasse die IQuestEventSubscriber implementeert
  2. Registreren in DI en subscriben bij de publisher in Program.cs

De QuestService en andere subscribers hoeven niet te worden aangepast.

8. Pattern Samenwerking

De kracht van dit project zit in hoe de 6 patterns samenwerken in de quest completion flow:

1. Quest wordt aangemaakt Strategy: IPricingStrategy berekent de prijs (Fixed/Auction/Dynamic) 2. Quest wordt weergegeven Decorator: UrgentDecorator + FeaturedDecorator + TeamDecorator stapelen tags, XP-bonussen en gold-multipliers 3. Quest wordt voltooid Observer: QuestEventPublisher.Notify() stuurt event naar 3 subscribers: 3a. XpCalculatorSubscriber Voegt XP en Gold toe, berekent level-up 3b. AchievementCheckerSubscriber Interpreter: AchievementRuleParser parsed DSL regels naar expression tree Evalueert regels tegen freelancer stats Flyweight: BadgeFactory.GetBadge() retourneert gedeeld badge-object Bridge: AchievementAlert + EmailSender stuurt notificatie 3c. LeaderboardUpdaterSubscriber Update leaderboard scores

Elk pattern lost een specifiek probleem op, en samen vormen ze een flexibel, uitbreidbaar systeem waar nieuwe pricing modellen, quest modifiers, notificatiekanalen, achievement regels en event reacties kunnen worden toegevoegd zonder bestaande code te wijzigen.
